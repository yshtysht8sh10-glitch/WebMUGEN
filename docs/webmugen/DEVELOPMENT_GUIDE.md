# WebMUGEN Development Guide

Updated: 2026-06-28

This is the first document to read before changing WebMUGEN. It records the project goals, compatibility priorities, file ownership rules, and expected development workflow.

## Project Goal

WebMUGEN aims to run WinMUGEN-era characters in the browser with high compatibility.

The target is not only to make one sample character move. The target is to preserve the behavior of real WinMUGEN character assets as much as possible.

Compatibility comes before convenience.

## Core Principles

1. Preserve WinMUGEN compatibility assets.
2. Keep common control flow visible in MUGEN-like text files when practical.
3. Prefer fixing loader, trigger evaluation, controller execution, physics, or runtime order over changing character/common assets.
4. Track compatibility one item per row in the matrix.
5. Keep tests and documentation aligned with every behavior change.

## File Ownership Rules

### `public/chars/common1.cns`

Do not edit this file as part of engine fixes.

`common1.cns` is treated as an external WinMUGEN compatibility asset. Many characters are authored with assumptions about the common state file. Replacing, minimizing, or regenerating it changes the compatibility baseline and makes behavior difficult to reproduce.

If crouch, walk, jump, or landing breaks after restoring `common1.cns`, fix the engine path instead:

- loader merge behavior
- command route merging
- trigger evaluation
- controller execution
- runtime state order
- physics integration
- animation timing

Do not solve that class of bug by rewriting `common1.cns`.

### `public/chars/common.cmd`

This file may be edited freely.

`common.cmd` is WebMUGEN's common control surface. When WinMUGEN-like common control logic can be expressed as CMD/CNS-style `Statedef -1` or `Statedef -2` routing, prefer putting it here instead of hiding it inside TypeScript fallback code.

Examples:

| Control logic | Preferred place |
|---|---|
| Crouch entry | `common.cmd` route to `State 10` |
| Walk forward | `common.cmd` route to `State 20` |
| Walk back | `common.cmd` route to `State 21` |
| Jump start | `common.cmd` route to `State 40` |
| Dash forward | `common.cmd` route to `State 100` when practical |
| Back dash | `common.cmd` route to `State 105` when practical |
| Landing transition | `common.cmd` or runtime support depending on expressiveness |
| Guard entry | `common.cmd` when trigger/controller support is sufficient |

TypeScript still implements parsing, trigger evaluation, controller execution, physics, collision, and rendering. But control rules that can be expressed in MUGEN-like text should be visible in `common.cmd`.

## Loader Policy

The loader must preserve ownership boundaries.

| Source | Loader behavior |
|---|---|
| Character CNS | Primary state source. |
| Character CMD | Primary command definitions and character-specific command routes. |
| `/chars/common1.cns` | Load if present; merge only missing states/routes. Never replace with embedded CNS. |
| `/chars/common.cmd` | Load after `common1.cns`; merge common command definitions/routes without replacing character/common1 State bodies. Editable control surface. |
| Embedded common CMD fallback | Allowed only when `/chars/common.cmd` is unavailable. Minimal safety net. |
| Embedded common CNS fallback | Not allowed. |

## Command Route Priority

Common movement must remain available when a character uses a direction as a modifier.

Example:

```cns
[State -1, Throw]
type = ChangeState
triggerall = command = "y"
trigger1 = command = "holdfwd"
value = 800
```

This does not mean `holdfwd` is the character's walking route. It means `holdfwd` modifies the `y` command. The common `holdfwd -> State 20` route should remain.

A character overrides a common direction route only when that direction is the character's primary route command.

Example:

```cns
[State -1, Character Jump]
type = ChangeState
triggerall = command = "holdup"
trigger1 = ctrl
value = 41
```

This may override common `holdup -> State 40`.

## Runtime Policy

TypeScript should behave as the execution engine, not as the hidden rulebook.

Runtime responsibilities:

1. Resolve input into command names.
2. Execute `Statedef -3`, `-2`, `-1`, and current state in the correct order as support improves.
3. Evaluate triggers faithfully.
4. Execute state controllers faithfully.
5. Apply state entry headers (`type`, `movetype`, `physics`, `anim`, `ctrl`).
6. Apply physics and ground handling.
7. Resolve facing, push, collision, hit, and recovery behavior.
8. Render the resulting state.

When behavior is missing, determine which layer failed before changing assets.

## Debugging Order

For movement, state, and hit bugs, inspect the pipeline in this order:

1. Input keys
2. Resolved command names
3. Loaded command routes in `Statedef -1`
4. Trigger evaluation
5. State transition
6. State entry header application
7. Controller execution
8. Physics step
9. Facing / push / collision
10. Hit resolution
11. Animation timing
12. Rendering

Do not start by editing `common1.cns`.

## Compatibility Matrix Rules

The compatibility matrix must remain explicit.

1. One item per row.
2. Do not combine multiple triggers, controllers, operators, redirects, states, or CMD features into one row.
3. Use four statuses only:
   - `Complete`
   - `Partial`
   - `Unsupported`
   - `Untested`
4. Mark `Complete` only when covered by focused tests or confirmed app integration.
5. Runtime shims and safe no-op recognition are `Partial`, not `Complete`.
6. Update both Markdown and HTML matrices when compatibility status changes.

Every State Controller should be tracked independently.

Examples:

| Controller | Tracking rule |
|---|---|
| `ChangeState` | One row. |
| `ChangeAnim` | One row. |
| `VelSet` | One row. |
| `VelAdd` | One row. |
| `PosSet` | One row. |
| `PosAdd` | One row. |
| `Turn` | One row. |
| `ScreenBound` | One row. |
| `HitDef` | One row. |
| `Helper` | One row. |
| `Explod` | One row. |

Do not group controllers as broad buckets such as `movement controllers` or `visual controllers` in the matrix item column.

Examples:

| Trigger | Tracking rule |
|---|---|
| `Time` | One row. |
| `StateNo` | One row. |
| `StateType` | One row. |
| `MoveType` | One row. |
| `Ctrl` | One row. |
| `Command` | One row. |
| `Pos X` | One row. |
| `Pos Y` | One row. |
| `Vel X` | One row. |
| `Vel Y` | One row. |
| `P2BodyDist X` | One row. |
| `P2BodyDist Y` | One row. |

Do not group triggers with slashes, ranges, or comma-separated lists.

## Development Workflow

Preferred order for compatibility work:

1. Identify the missing behavior.
2. Locate the expected WinMUGEN asset path or documented behavior.
3. Update or check the matrix entry.
4. Add or update a focused test.
5. Implement the smallest engine/runtime/parser change that supports the behavior.
6. Update docs if the design rule changed.
7. Run the relevant tests and, when possible, the full suite.

For UI/app behavior, also verify the active app path. Previous fixes landed in test-only paths, so make sure `WebMugenApp` actually calls the modified runtime path.

## Git Policy

Use small commits.

Good commit examples:

```text
fix: keep common movement routes when character uses direction modifiers
test: cover common crouch route loading
docs: document common CNS loading policy
```

Avoid mixing unrelated changes, especially code changes and broad matrix rewrites, unless the implementation directly changes compatibility status.

## Current High Priority Areas

1. Crouch transition reliability (`holddown -> State 10`).
2. Walk forward reliability (`holdfwd -> State 20`).
3. Walk back reliability (`holdback -> State 21`).
4. Facing / turn behavior.
5. Player push and collision so characters cannot pass through each other.
6. HitDef and attack contact reliability.
7. State `-3` execution.
8. Air physics and `State 51`/`State 52` sequencing.
9. Arithmetic expression parser support.
10. True target/enemy/helper lookups.

## Non-Negotiable Rules

1. Do not edit `public/chars/common1.cns` unless the explicit task is to update that compatibility asset.
2. Do not reintroduce embedded common CNS replacement text.
3. `public/chars/common.cmd` is the preferred editable surface for common control routing.
4. Keep common behavior visible as MUGEN-like data when practical.
5. Keep the compatibility matrix current.
6. Prefer compatibility and reproducibility over short-term fixes.
